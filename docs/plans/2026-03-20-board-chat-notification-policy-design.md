# Board Chat Notification Policy Design

**Date:** 2026-03-20
**Status:** Validated

## Problem

Board activity can generate too many external chat notifications for mobile users. The current chat architecture treats Telegram and other providers as delivery mechanisms for all outbound questions and notifications, with no board-level control over which categories should reach a phone.

This creates two product problems:

- Users get noisy phone notifications for routine workflow chatter such as builder updates.
- Board settings are likely to grow, and the current settings page will become long and harder to scan.

The desired behavior is board-scoped and delivery-specific:

- Potato Cannon UI remains unchanged.
- Conversation history remains unchanged.
- External chat delivery becomes configurable per board.
- The solution must be provider-agnostic and must not put mute logic into Telegram-specific code.

## Goal

Add a board-level external chat notification policy that lets users choose how much board activity should reach their phones, while keeping all messages visible in Potato Cannon. Add collapsible sections to the board settings page so this and future board-level settings remain manageable.

## Non-Goals

- No global or project-wide mute policy in this feature.
- No provider-specific settings or Telegram-only behavior.
- No suppression of messages from Potato Cannon UI, SSE, or stored conversation history.
- No digesting, bundling, or delayed delivery in the first version.
- No text-pattern or regex-based mute rules in the first version.

## User Intent

Users want external chat to be interruptive only when it is valuable. Some boards may want full delivery at all times, while others should stay quiet except for important signals. Because the same questions and notifications still exist inside Potato Cannon, users can safely mute external delivery without losing access to information.

## Options Considered

| Option | Description | Pros | Cons | Verdict |
|---|---|---|---|---|
| Text or pattern mute rules | Store phrases or regex-like rules and suppress matching outbound text | Fast to prototype | Brittle, hard to explain, easy to break when message text changes | Reject |
| Typed category toggles | Classify outbound messages into stable categories and suppress by board policy | Clean architecture, provider-agnostic, easy to evolve | Requires explicit category plumbing | Accept |
| Delivery levels only | Use broad levels like `all`, `important`, `quiet` without category controls | Simple UI | Too coarse for “mute builder updates” | Reject |
| Preset + advanced toggles | Users pick a preset, then optionally fine-tune category toggles | Best mix of usability and control | Slightly more UI work | Accept |

## Product Decisions

- **Scope:** Policy is stored at the board level.
- **Delivery boundary:** Policy affects only outbound chat provider delivery.
- **Provider architecture:** Filtering happens before provider fan-out in shared chat logic.
- **Provider settings:** No per-provider toggles in this feature.
- **Default preset for new boards:** `All notifications`.
- **Important Only preset:** `questions + critical + pm_alerts`.
- **Questions:** Users may mute questions externally because they remain visible in Potato Cannon.
- **Settings UX:** Use collapsible sections on the board settings page.

## Proposed Model

Add a new board-scoped policy object alongside the existing PM config:

```ts
type BoardNotificationPreset =
  | "all"
  | "important_only"
  | "questions_only"
  | "mute_all";

type ChatNotificationCategory =
  | "builder_updates"
  | "pm_alerts"
  | "lifecycle_events"
  | "questions"
  | "critical";

interface ChatNotificationPolicy {
  preset: BoardNotificationPreset;
  categories: Record<ChatNotificationCategory, boolean>;
}
```

Extend `BoardSettings` to store both:

- `pmConfig`
- `chatNotificationPolicy`

Defaults when no board row exists:

- `preset = "all"`
- all category toggles enabled

## Category Definitions

| Category | Meaning | Examples |
|---|---|---|
| `builder_updates` | Routine progress chatter from workflow execution | task started, task completed, builder progress messages |
| `pm_alerts` | Signals generated by PM monitoring logic | stuck ticket, dependency unblocked, session crash, ralph failure |
| `lifecycle_events` | Status changes in session or ticket lifecycle | paused, resumed, blocked, phase transition |
| `questions` | Human-response-needed outbound prompts | `chat_ask` questions, option prompts |
| `critical` | High-signal action-needed notifications | explicit “needs human attention now” warnings |

The first version should classify only known, structured notification sources. Anything not explicitly categorized should be treated conservatively by the shared chat layer rather than silently dropped.

## Delivery Architecture

### Current architecture

- Messages are persisted to the conversation store.
- Real-time updates are emitted to the UI.
- `ChatService` fans outbound messages to all active providers.
- Providers like Telegram and Slack are responsible only for thread lookup and transport.

### Proposed architecture

Keep persistence and UI behavior unchanged. Add filtering only in the outbound path:

1. A message is created with a stable chat notification category.
2. The message is persisted to conversation history as it is today.
3. The message is emitted to the UI as it is today.
4. Before provider fan-out, shared chat logic resolves the board/workflow for the current ticket or brainstorm.
5. Shared chat logic resolves the board’s notification policy.
6. If the category is enabled, the message is sent to active providers.
7. If the category is disabled, provider delivery is skipped and the UI/history still remain intact.

This keeps all mute decisions in shared board-aware code and keeps providers transport-only.

## Classification Strategy

Do not classify by message text.

Instead, extend shared outbound chat metadata so a question or notification carries a provider-agnostic category. Likely changes:

- `OutboundMessage` gains `category?: ChatNotificationCategory`
- question paths default to `questions`
- existing workflow and session notify paths assign explicit categories
- PM alert paths assign `pm_alerts`

This preserves provider independence and makes future providers free riders on the same policy.

## Board Settings UX

The board settings page should be reorganized into collapsible sections:

- `Project Manager`
- `Phone Notifications`
- `Advanced`

`Phone Notifications` contains:

- A short explanation that Potato Cannon still shows all messages even when phone delivery is muted
- A preset selector with:
  - `All notifications`
  - `Important only`
  - `Questions only`
  - `Mute all external chat`
- An expandable advanced controls area with category-level toggles

Preset semantics:

| Preset | Enabled categories |
|---|---|
| `All notifications` | all categories |
| `Important only` | `questions`, `critical`, `pm_alerts` |
| `Questions only` | `questions` only |
| `Mute all external chat` | none |

Advanced toggles should directly reflect the resolved category booleans. Selecting a preset writes the booleans under the hood. If a user customizes toggles afterward, the UI can either show a custom state or leave the preset selected until save; implementation can decide later, but the stored booleans are the source of truth.

## Data And API Changes

Shared types:

- extend `packages/shared/src/types/board-settings.types.ts`
- export notification preset/category/policy types

Backend:

- extend `apps/daemon/src/stores/board-settings.store.ts` to merge and resolve policy defaults
- extend `apps/daemon/src/server/routes/board-settings.routes.ts` validation and payloads
- add any required DB migration for new `board_settings` storage shape

Frontend:

- update `apps/frontend/src/components/configure/BoardSettingsPage.tsx`
- ensure the page uses board settings API-backed data rather than local-only storage for board-specific values
- add collapsible section components or extend `SettingsSection`

Chat system:

- extend `apps/daemon/src/providers/chat-provider.types.ts` for provider-agnostic category metadata
- add filtering in `apps/daemon/src/services/chat.service.ts`
- resolve workflow id from `ticket` or `brainstorm` records before applying board policy
- update message creation paths in session/PM/chat tools to assign categories

## Edge Cases

- **Muted questions:** Question remains visible in Potato Cannon and stored normally, but no phone push is sent.
- **Unknown category:** Do not silently suppress. Prefer explicit classification for structured sources and safe fallback behavior for anything uncategorized.
- **No board overrides yet:** Use default preset and toggles without requiring a saved settings row.
- **Future providers:** They inherit the same filtering automatically because the policy lives above provider transport.
- **Brainstorms vs tickets:** The policy is board-scoped, so any outbound chat associated with a workflow/board should resolve through that workflow’s board settings.

## Risks

| Risk | Why it matters | Mitigation |
|---|---|---|
| Incomplete classification | Some noisy messages might bypass policy | Audit known `chatService.notify` and `chat_ask` call sites and add explicit categories |
| Wrong filtering layer | Putting logic in Telegram would block future provider reuse | Keep logic only in shared board-aware chat service path |
| Settings page sprawl | More board settings will keep accumulating | Add collapsible sections now as part of the same feature |
| UI/API mismatch | Board settings page appears partially local today | Use this feature to finish API-backed board settings wiring |
| Silent data loss perception | Users may think mute hides messages entirely | Add clear explanatory copy in `Phone Notifications` section |

## Testing Strategy

- Unit test board settings default resolution and merge behavior for notification policy.
- Unit test board settings route validation and returned payload shape.
- Unit test chat service filtering so muted categories are stored/emitted but not sent to providers.
- Unit test preset-to-toggle mapping.
- Unit test category toggle persistence and reload.
- UI test collapsible board settings sections and phone notification preset behavior.
- Regression test that muting a board does not change Potato Cannon activity feed behavior.

## Success Criteria

- Users can configure external chat noise level per board.
- Builder updates can be muted without changing Potato Cannon UI history.
- Questions can be muted externally and still remain answerable in Potato Cannon.
- No provider-specific mute logic is added to Telegram or future providers.
- Board settings remain navigable with collapsible sections.

## Recommendation

Implement this as a board-scoped, provider-agnostic `preset + advanced toggles` policy enforced in shared outbound chat delivery. Keep persistence and UI untouched, keep providers transport-only, and treat collapsible board settings sections as part of the same feature so the settings page scales cleanly.
