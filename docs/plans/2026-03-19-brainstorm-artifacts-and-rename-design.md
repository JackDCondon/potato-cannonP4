# Brainstorm Artifacts & Rename Design

**Date:** 2026-03-19
**Status:** Approved
**Problem:** Brainstorm sessions produce rich context (plans, decisions, ticket rationale) but none of it is visible or durable. The plan_summary is a 200-word text field never surfaced in the UI. Conversation history is buried in chat scroll. Humans can't review the brainstorm's output, and agents get only a brief summary.

## Approach: File-Based Brainstorm Artifacts + Editable Names

Store brainstorm artifacts as markdown files in the existing brainstorm directory. Add an Artifacts tab to the brainstorm panel for human review. The brainstorm agent auto-generates a `plan.md` artifact after creating tickets. Also add the ability to rename brainstorms so they have clean epic-style names instead of auto-generated summaries.

### Why Not DB-Backed Artifacts?

- The ticket artifact system (DB table + versioning + manifest) is heavyweight for brainstorm needs
- Brainstorm artifacts are primarily read-only markdown documents
- Simple files avoid migration, store changes, and schema coupling
- Can always migrate to DB-backed later if versioning/metadata becomes needed

## 1. Storage

### File System Layout

```
~/.potato-cannon/projects/{projectId}/brainstorms/{brainstormId}/
├── artifacts/
│   └── plan.md          ← Auto-generated plan document
└── pending-*.json       ← Existing IPC files
```

The `getBrainstormFilesDir()` function in `config/paths.ts` already resolves the parent path. We create an `artifacts/` subdirectory inside it.

### Plan Document Structure

Generated by the brainstorm agent:

```markdown
# [Epic Name] — Plan

## Goal
[What we're building and why — 1-2 paragraphs]

## Ticket Breakdown
| ID | Title | Complexity | Handles |
|----|-------|------------|---------|
| ... | ... | ... | ... |

## Architecture Decisions
[Key technical choices made during brainstorming]

## Scope Boundaries
[What's explicitly NOT included and why]
```

### API

One new endpoint:

- `GET /api/brainstorms/:project/:id/artifacts` — reads files from the artifacts directory

Response:
```typescript
{
  artifacts: Array<{
    filename: string    // e.g. "plan.md"
    content: string     // file contents
    updatedAt: string   // file mtime as ISO 8601
  }>
}
```

No write endpoint — artifacts are created by MCP tools during brainstorm sessions.

## 2. MCP Tools

### New: `save_brainstorm_artifact`

Writes a markdown file to the brainstorm artifacts directory.

```typescript
{
  name: "save_brainstorm_artifact",
  description: "Save an artifact file to the current brainstorm",
  inputSchema: {
    type: "object",
    properties: {
      filename: {
        type: "string",
        description: "Filename for the artifact (e.g. plan.md)"
      },
      content: {
        type: "string",
        description: "File content (markdown)"
      }
    },
    required: ["filename", "content"]
  },
  scope: "session"
}
```

**Behavior:**
- Validates `ctx.brainstormId` exists (brainstorm session only)
- Sanitizes filename (alphanumeric, hyphens, dots only — no path traversal)
- Creates `artifacts/` directory if needed
- Writes file, overwrites if exists
- Returns `{ saved: true, filename, path }`

### New: `rename_brainstorm`

Sets a clean, descriptive name for the brainstorm/epic.

```typescript
{
  name: "rename_brainstorm",
  description: "Set a clear, concise name for this brainstorm/epic",
  inputSchema: {
    type: "object",
    properties: {
      name: {
        type: "string",
        description: "Short epic-style name (3-8 words)"
      }
    },
    required: ["name"]
  },
  scope: "session"
}
```

**Behavior:**
- Validates `ctx.brainstormId` exists
- Calls `PUT /api/brainstorms/:project/:id` with `{ name }`
- Returns `{ renamed: true, name }`

### Enhanced: `set_plan_summary`

Existing behavior unchanged. **Addition:** After storing the summary on the DB row, also write `artifacts/plan.md` with a basic formatted version of the summary. This provides a fallback — even if the agent doesn't call `save_brainstorm_artifact`, the plan summary still appears in the Artifacts tab.

The auto-generated `plan.md` from `set_plan_summary` is overwritten if the agent subsequently calls `save_brainstorm_artifact` with a richer version.

## 3. Frontend UI

### Brainstorm Detail Panel — Add Tabs

Current: `BrainstormDetailPanel` renders `BrainstormChat` directly.
New: Wrap in a tab layout with Chat and Artifacts tabs.

```
┌─────────────────────────────────────┐
│ 🔆 [editable brainstorm name] [X]  │
│                                     │
│  [ Chat ]  [ Artifacts ]            │
│ ─────────────────────────────────── │
│                                     │
│  (tab content here)                 │
│                                     │
└─────────────────────────────────────┘
```

**Chat tab** — existing `BrainstormChat` component, unchanged.

**Artifacts tab** — new `BrainstormArtifactsTab` component:
- Fetches `GET /api/brainstorms/:project/:id/artifacts`
- If artifacts exist: renders list of artifact cards, click to expand/view markdown
- If no artifacts: shows empty state message
- Markdown rendered with the same `renderMarkdown()` utility used in ticket descriptions
- Read-only (no editing in MVP)

### Editable Brainstorm Name

In the panel header:
- Brainstorm name text becomes click-to-edit (inline input)
- On blur or Enter: calls `PUT /api/brainstorms/:project/:id` with `{ name: newName }`
- Query invalidation updates `BrainstormCard` on the board
- Escape cancels edit

### API Client & Query Hook

```typescript
// client.ts
getBrainstormArtifacts: (projectId: string, brainstormId: string) =>
  fetchJson<BrainstormArtifactsResponse>(`/api/brainstorms/${enc(projectId)}/${brainstormId}/artifacts`)

// queries.ts
export function useBrainstormArtifacts(projectId: string | null, brainstormId: string | null) {
  return useQuery({
    queryKey: ['brainstorm-artifacts', projectId, brainstormId],
    queryFn: () => api.getBrainstormArtifacts(projectId!, brainstormId!),
    enabled: !!projectId && !!brainstormId,
  })
}
```

## 4. Agent Prompt Changes

### Brainstorm Agent (`brainstorm.md`)

Add after existing ticket creation instructions:

> After creating all tickets for a multi-ticket plan:
> 1. Call `rename_brainstorm` with a clear, concise epic name (e.g. "User Authentication System" not "Discussing auth options").
> 2. Call `save_brainstorm_artifact` with filename `plan.md` containing a structured plan document with: Goal, Ticket Breakdown (table), Architecture Decisions, and Scope Boundaries.
> 3. Call `set_plan_summary` with a brief 100-200 word summary (this is injected into agent prompts — keep it concise).

The order matters — rename first (so the plan doc can reference the epic name), then the full artifact, then the compact summary.

## 5. Scope Boundaries

### In Scope

- File-based brainstorm artifact storage
- `save_brainstorm_artifact` MCP tool
- `rename_brainstorm` MCP tool
- Enhanced `set_plan_summary` with auto-generated plan.md fallback
- Brainstorm panel tabs (Chat / Artifacts)
- Markdown artifact viewer (read-only)
- Inline brainstorm name editing
- Brainstorm agent prompt updates

### Not in Scope

- Artifact editing in UI
- Artifact versioning for brainstorms
- Per-ticket brief generation
- Ticket list in artifacts tab
- Artifact search/filtering
- DB-backed brainstorm artifacts

## 7. Design Decisions

| Decision | Rationale |
|----------|-----------|
| File-based, not DB-backed | Brainstorm artifacts are simple markdown files; avoid migration and schema coupling |
| "Artifacts" tab, not "Plan" tab | Mirrors ticket artifact pattern; supports any markdown file, not just plans |
| Auto-generate plan.md from set_plan_summary | Fallback ensures artifacts appear even without explicit save_brainstorm_artifact call |
| rename_brainstorm as MCP tool | Agent can set a clean name at the right moment; humans can also edit inline |
| Read-only artifact viewer | Keep MVP simple; editing can come later |
| No artifact versioning | Files overwrite; keep complexity minimal for brainstorm context |
